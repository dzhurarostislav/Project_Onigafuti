he Onigari Architecture Manifest (V 1.0)Project Goal: High-performance 2D evolutionary simulation using ECS-lite and NumPy.1. Technical Stack & StandardsLanguage: Python 3.10+Core Libraries: numpy (math), pygame (rendering).Performance Paradigm: Data-Oriented Design. Logic must be separated from Data.Vectorization: Grid operations must be vectorized using NumPy (no pixel-by-pixel loops).2. The Data Contract (The World State)All logic systems must interact with a single World object containing:nutrients (float32): Potential energy in soil.vegetation (float32): Ready-to-eat energy.agent_map (int32): Grid storing Agent IDs (-1 = empty).scent_map (float32): Simplified gradient for navigation.leak_buffer (float): Global energy pool from boundary losses.sun_pos (float[2]): Current center of the roaming oasis.3. Simulation Logic RulesThe Pump: Energy "leaks" at boundaries $\rightarrow$ goes to leak_buffer $\rightarrow$ slowly "drips" back into the grid at sun_pos.The Drift: sun_pos moves via slow Brownian motion ($V_{sun} \ll V_{agent\_min}$).The Threshold: Diffusion stops if a cell's nutrient value is below a small $\epsilon$ to prevent absolute vacuum.The Fairness: Systems must shuffle agent processing order every tick to prevent index-based priority (race conditions).4. Integration ProtocolSystems are stateless functions or classes that take World and list[Agents] as input.Rendering is read-only. It must NEVER mutate the World state. "The state is a single source of truth. If your logic produces a persistent effect on the world, it MUST be written into the World instance matrices. Never invent variables outside the agreed World schema."
